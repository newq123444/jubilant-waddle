# SmartRota hardening, CSV rota import, SBAR AI fallback, and sidebar search

Four disparate changes bundled in a single commit: the SmartRota page gains error states and a CSV upload feature; the SBAR handover flow gets a structured fallback when the Anthropic API key is absent; and the App sidebar gets a text filter for its 60+ nav items. The approach is pragmatic—each feature is self-contained, builds on existing patterns (TanStack Query mutations, multer middleware, existing controller structure), and both frontend and backend compile cleanly.

**Watch for:**
- CSV upload processes rows sequentially with one DB query per row for staff lookup and one for profile lookup—O(2n) individual queries with no batching or transaction wrapping (confirmed).
- The `uploadRotaCsv` endpoint accepts `.xlsx` in the file filter but only implements CSV (comma-split) parsing—uploading a real Excel binary will produce garbage rows silently (confirmed).
- The AI fallback sentinel comparison `result === AI_NOT_CONFIGURED` is brittle: if the exported constant value changes or the string comparison fails due to import aliasing, the fallback path is skipped and the raw sentinel string gets stored as the SBAR situation text (confirmed).

## High-level view

The SmartRota page now handles three failure modes gracefully: empty template lists show a loading/error state instead of crashing on undefined array access, the generation mutation surfaces errors in an inline banner, and the constraints panel is hidden when its query fails.

The CSV upload feature spans the full stack: a new multer memory-storage config accepts CSV/XLSX files, a new `uploadRotaCsv` controller parses the buffer line-by-line, looks up staff by name, and upserts shifts. The implementation works for small files but has no transaction boundary—a failure midway through leaves partial data—and the per-row sequential queries will be slow for files with hundreds of rows.

The SBAR handover fallback introduces a `buildFallbackSbar` function that assembles all four SBAR sections from the raw database query results already fetched for the AI prompt. The sentinel-string comparison pattern is the weakest link in the control flow—the controller is coupled to exact string values exported by `ai.service.ts`.

The sidebar search is a controlled input that filters `NAV_ALL` by label substring match, hiding section headers during active search. It disappears when the sidebar is collapsed, but `searchQuery` state persists across collapse/expand cycles, which can leave a stale filter active.

<details>
<summary>Issues (7)</summary>

1. **XLSX accepted but not parsed** — The file filter allows `.xlsx` uploads, but the parser does `buffer.toString('utf-8').split(/\r?\n/)` which corrupts binary Excel files. Either reject XLSX in the filter or add a library like `xlsx`/`exceljs` to parse it.
2. **No transaction on CSV import** — Each shift insert is independent. A parse error on row 50 leaves rows 1–49 committed. Wrap the loop in a database transaction so the import is atomic.
3. **Sequential per-row queries** — Staff lookup (`SELECT ... WHERE LOWER(first_name)=...`) and profile lookup run once per CSV row. For a 200-row file this is 400+ round-trips. Batch staff name resolution before the insert loop.
4. **Sentinel comparison fragility** — `result === AI_NOT_CONFIGURED` depends on the exact string export. If `runAiOperation` is refactored to throw instead of returning sentinels, the fallback path silently breaks. A typed return union or error subclass would be safer.
5. **AI error-string stored as audit "success"** — When the Anthropic call throws, `runAiOperation` catches the error, sets `outputText = 'AI service temporarily unavailable: ...'`, but still logs `status: 'success'` to `ai_audit_log`. The audit record should reflect the failure.
6. **`startsWith` check for AI unavailability is loose** — `result.startsWith('AI service temporarily unavailable')` could match legitimate AI output that coincidentally starts with that phrase (unlikely but possible). Use a typed discriminated return instead.
7. **CSV name matching is brittle** — `staffName.split(/\s+/)` assigns `nameParts[0]` as first name and everything else as last name. Staff with middle names, double-barrelled first names, or name prefixes will fail to match silently.

## CSV upload: no transaction boundary and sequential queries

The `uploadRotaCsv` controller iterates over parsed CSV rows and, for each row, executes two SELECT queries (user lookup by name, profile lookup by user_id) and one INSERT/UPSERT:

```typescript
for (let i = 1; i < lines.length; i++) {
  // ... parse row ...
  const { rows: users } = await query(`SELECT u.id ... WHERE LOWER(first_name) = LOWER($2) AND LOWER(last_name) = LOWER($3)`, ...);
  const { rows: profiles } = await query(`SELECT id FROM staff_profiles WHERE user_id = $1 ...`, ...);
  await query(`INSERT INTO shifts ... ON CONFLICT ... DO UPDATE ...`, ...);
}
```

For a 100-row CSV this is 300 database round-trips executed sequentially. More critically, there is no transaction. If the server crashes or the request is aborted after row 60, 60 shifts are committed and 40 are lost—with no indication to the user about which rows persisted. The response `details` array communicates per-row status, but only if the request completes normally.

A `BEGIN`/`COMMIT` wrapper with `ROLLBACK` on error would make the operation atomic. Pre-loading all staff names into a Map before the loop would reduce lookups from O(n) to O(1).

## XLSX file handling gap

The `csvFileFilter` in `upload.ts` accepts both `text/csv` and `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`. The frontend file input also accepts `.csv,.xlsx`. However, the controller does:

```typescript
const content = req.file.buffer.toString('utf-8');
const lines = content.split(/\r?\n/).filter(line => line.trim() !== '');
```

XLSX is a ZIP-compressed XML format. Interpreting it as UTF-8 text produces binary garbage. The likely outcome is a 400 error about missing columns (binary won't match header names). The unlikely-but-worse outcome is random bytes that happen to include commas and newlines producing rows that pass validation but insert nonsense data into the shifts table.

Either remove `.xlsx` from the accepted types (and update the frontend `accept` attribute) or add proper XLSX parsing.

## AI service audit status mismatch

In `ai.service.ts`, when the Anthropic API call throws:

```typescript
} catch (err: any) {
  logger.error('AI operation failed:', err.message);
  outputText = `AI service temporarily unavailable: ${err.message}`;
}

// Always audit AI operations
await query(`INSERT INTO ai_audit_log ... VALUES (..., 'success')`, ...);
```

The status column is hardcoded to `'success'` regardless of whether the call actually succeeded. Operators querying `ai_audit_log WHERE status = 'error'` will never find transient API failures. Add a `let auditStatus = 'success'` variable that flips to `'error'` in the catch block.

## SBAR fallback sentinel pattern

The handover controller checks two conditions to decide whether to use the fallback:

```typescript
if (result === AI_NOT_CONFIGURED || result.startsWith('AI service temporarily unavailable')) {
  sections = buildFallbackSbar(...);
}
```

This couples the controller to the exact string format of the AI service's error output. If anyone changes the error message prefix in `ai.service.ts`, the fallback stops triggering and the raw error string gets stored as the SBAR situation text in the database. A cleaner contract would be for `runAiOperation` to return a discriminated union (`{ ok: true, text: string } | { ok: false, reason: string }`) or throw a typed error that the controller catches explicitly.

## Sidebar search state persists across collapse

The search input is hidden when `isDesktop && collapsed` is true, but `searchQuery` is not cleared when the sidebar collapses. If a user types a filter, collapses the sidebar, then expands it again, the filter is still active with the old query text. The nav items will be filtered with no obvious visual cue until the user notices the populated input. Clearing `searchQuery` when `collapsed` transitions to `true` would eliminate this edge case.
