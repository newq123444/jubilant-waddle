# Mobile responsiveness via CSS overrides and targeted component fixes

The CareVista frontend gains mobile and tablet responsiveness through a layered CSS strategy: attribute selectors override inline grid styles at breakpoints, tap targets grow to 44px on phones, the login page hides its branding panel on narrow screens, and two component files get minimal `className` or `flexWrap` additions to hook into the new CSS. The approach avoids refactoring dozens of inline-styled components by using CSS specificity (`!important`) to coerce their rendered DOM styles at runtime.

**Watch for:** The `[style*="grid-template-columns"]` attribute selector is a blunt instrument that will collapse *every* inline grid to single-column on phones, including grids (calendar views, data tables) where that may produce a broken layout (confirmed). Several major dashboard pages (Cleaning, Kitchen, Activities) use inline `gridTemplateColumns` extensively and receive no targeted responsive treatment beyond the global override (confirmed).

## High-level view

The responsiveness strategy is CSS-only with minimal React touch. Rather than converting inline styles to utility classes or a design system, the change adds attribute selectors that match against the browser's rendered `style` attribute string. This works because React serializes `gridTemplateColumns` as `grid-template-columns` in the DOM. The tradeoff is that every grid in the app — regardless of whether single-column makes sense for it — gets flattened on mobile. Pages like the 7-column calendar in Activities or the room-status grid in CleaningDashboard will likely need exemptions.

The Login page restructuring is well-scoped: a mobile header appears when the branding panel disappears, and the demo-account grid collapses at the narrowest breakpoint. The Carer Dashboard gets a named class specifically so its two-panel layout stacks below 1024px, avoiding the global selector's phone-only breakpoint.

The Residents page `flexWrap: 'wrap'` addition prevents horizontal overflow, but the container's `gap: 14` means wrapped items produce vertical spacing whose visual intent may be ambiguous at intermediate widths.

<summary>Issues (4)</summary>

1. **Global grid override too aggressive** — `[style*="grid-template-columns"]` at 639px collapses every inline grid including data tables, calendars, and multi-column layouts where single-column is incorrect. Add opt-out class (e.g., `.no-mobile-collapse`) or narrow the selector.
2. **Tablet selector matches string fragments** — `[style*="repeat(4"]` will match any element whose style attribute *contains* the string "repeat(4" anywhere, including `repeat(4,` in shorthand gap definitions or unrelated properties. False positives are unlikely today but fragile.
3. **No coverage for CleaningDashboard, KitchenDashboard, Activities** — These pages have 5-column, 3-column, and 7-column grids with dense data. The global override will stack all columns into a single column on phones, likely making the UI unusable. They need either explicit responsive classes or opt-out markers.
4. **Empty rule declarations** — `.login-left { }`, `.login-right { }`, `.login-demo-grid { }` are no-ops on desktop but exist solely to prevent the browser from discarding the selectors before the media query variants. This is harmless but could confuse future maintainers — a comment explaining the pattern would help.

## Attribute selector as responsive escape hatch

The core technique — `[style*="grid-template-columns"] { grid-template-columns: 1fr !important; }` — exploits the fact that React serializes camelCase style props into their hyphenated CSS equivalents in the DOM. An earlier version of this code used `[style*="gridTemplateColumns"]` which would never match; the fix to the hyphenated form is correct.

The concern is scope. There are at least 25 uses of inline `gridTemplateColumns` across the frontend (Activities calendar with `repeat(7, 1fr)`, Cleaning dashboard with `repeat(5, 1fr)` and `2fr 1fr`, Kitchen dashboard with `repeat(4, 1fr)`, DailyPurposePlanner with `repeat(auto-fill, ...)`). All of these will collapse to a single column at 639px. For data grids this produces a vertical stack hundreds of items tall; for the calendar it turns a week view into a day view. The fix needs either an exclusion mechanism or more targeted selectors that match only "generic layout" grids.

The tablet selector (`[style*="repeat(4"]`, `[style*="repeat(5"]`) is more restrained — it only collapses 4- and 5-column grids to 2 columns — but matches against raw string fragments in the style attribute, making it fragile if any future element happens to contain that substring in a different context.

## Pages with no responsive treatment beyond the global override

CleaningDashboard uses `repeat(5, 1fr)` for KPI cards, `2fr 1fr` for the main grid, and `1fr 1fr 1fr` for a third row. KitchenDashboard uses `repeat(4, 1fr)` for meal service cards and `2fr 1fr 1fr 1fr` for temperature logs. Activities uses `repeat(7, 1fr)` for the calendar. None of these receive a named class or targeted breakpoint. On phones, the global selector collapses all of them to single-column. The 7-day calendar becoming a single column is particularly problematic — each day cell will take full width, requiring users to scroll through 35+ cells vertically to see a week.
