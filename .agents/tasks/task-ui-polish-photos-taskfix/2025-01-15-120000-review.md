# Crash fix, UI maturation, and resident avatars

This change addresses three issues in a single pass: a crash when opening clinical forms from the carer dashboard, overly saturated/playful visual styling, and missing resident profile photos on the task boards. The crash was caused by `ClinicalForm` referencing `activeResident.first_name` â€” a variable from the parent component's scope â€” instead of its own `resident` prop. The styling overhaul shifts the entire design system toward a more muted, clinical palette with reduced font weights and subtler shadows. Resident photos (36px circular avatars with initials fallback) are added to both CarerDashboard and TaskBoard.

**Watch for:** The `residentPhoto` URL helper duplicates logic already present in ResidentDetail.tsx and Staff.tsx, creating a maintenance surface that could drift (confirmed). A second `activeResident.first_name` reference at line 1090 in CarerDashboard.tsx was *not* fixed, though it's in the parent component scope where `activeResident` is declared, so it's correct â€” but worth noting for anyone investigating crashes in that area (confirmed, not a bug).

## High-level view

The crash fix is a one-character change (prop name correction) that eliminates a ReferenceError inside `ClinicalForm`. The component receives `resident` as a prop but the template interpolation referenced `activeResident`, which exists only in the parent `CarerDashboard` function body. This would crash on every form open since `activeResident` is simply undefined in that scope.

The CSS overhaul is a comprehensive design-token refresh: darker primary palette, reduced border-radii (10pxâ†’8px, 14pxâ†’12px), lighter shadows, lower font-weights across headings and labels, and a new `--font-heading` variable. Dark mode gets its own shadow scale and corrected hover highlights. The badge component shifts from pill-shaped (20px radius) to a squared tag style (6px radius). The specificity of certain dark-mode selectors could create subtle ordering issues if the stylesheet grows.

The photo feature adds a `useResidents` query to TaskBoard (the hook was already in use on CarerDashboard) and looks up each resident by ID at render time to get photo URLs. Both pages now render a 36px avatar before each resident's name, falling back to a two-letter initials bubble. This means an extra network round-trip to fetch all residents â€” the task API already joins resident data but doesn't include `photo_url`.

## ClinicalForm crash: scoping error

`ClinicalForm` is a standalone function component declared at line 354 of CarerDashboard.tsx. It receives `resident` as a destructured prop. At line 772, the placeholder text interpolated `activeResident.first_name` â€” a variable that only exists in the `CarerDashboard` default export (declared at line 856). In strict mode this throws a ReferenceError and blanks the page. The fix changes it to `resident.first_name`, which is the prop in scope.

## Duplicated photo URL resolution

Four files now contain nearly identical `residentPhoto` / `photoSrc` helpers:

```
TaskBoard.tsx:        residentPhoto(url) â†’ string | null
CarerDashboard.tsx:   residentPhoto(url) â†’ string | null
ResidentDetail.tsx:   photoSrc(url) â†’ string
Staff.tsx:            photoSrc(url) â†’ string
```

They all strip `/api` from `VITE_API_URL` to get the backend origin, then prefix relative paths. The return types differ slightly (nullable vs empty string) but the intent is identical. If the URL scheme changes (e.g., photos move to a CDN), four files need updating.

## Additional resident query in TaskBoard

TaskBoard now calls `useResidents({ active: true })` to get photo URLs. The task list API already returns `resident_name` and `room_number` per task, suggesting the backend joins against the residents table. Adding `photo_url` to that join would eliminate a separate query that fetches *every* resident just to look up photos. This is an extra network round-trip on every TaskBoard mount â€” not a correctness issue, but inefficient for large facilities.

## Dark-mode gaps in avatar fallback

The initials-fallback avatar uses inline styles with hardcoded colors:

```tsx
background: '#e0e7ff', border: '2px solid #c7d2fe', color: '#4338ca'
```

These light-indigo tones will look out of place against the dark surface (`#1a1d27`) since they bypass the CSS variable system entirely.

## Global card hover elevation

The new rule `.card:hover { box-shadow: var(--shadow) }` applies to every element with `.card`, including static information panels, stat summary cards, and form containers. Cards that aren't clickable or navigable shouldn't suggest interactivity via hover elevation changes â€” it creates a false affordance.
