# Three operational roles, four reusable widgets, and v1 fixes (v2)

This change adds cleaning, kitchen, and maintenance roles to a care home management app (CareVista), each with a dedicated ~300-line dashboard, three demo login accounts, backend seed data, and role enum expansion. Four reusable widgets (WeatherWidget, FacilityMap, StaffWellnessWidget, LiveActivityFeed) are integrated across dashboards. The v2 revision addresses all five issues raised in v1: StaffWellnessWidget is now rendered in all three operational dashboards, Care Notes and Incidents nav items have explicit role lists excluding operational staff, the Report Concern form has controlled state and a working submit handler, room data is aligned between FacilityMap and CleaningDashboard (24 rooms, same residents), and `useDashboard` data is wired into the KPI sections.

**Watch for:** Residents, Policies, and Family Portal nav items still use `roles: null`, meaning operational staff see them — Residents in particular exposes care data (confirmed). FacilityMap and CleaningDashboard use semantically different status vocabularies for the same rooms, which could confuse future maintainers even though the visual output is coherent (confirmed). LiveActivityFeed's `useMemo` depends on `new Date()` in its creation but lacks a proper dependency, meaning it will only re-compute on remount (likely).

**Verdict**: APPROVED

## High-level view

All five v1 issues are resolved. The nav visibility fix is narrowly scoped — Care Notes, Medications, and Incidents now exclude operational roles explicitly, but Residents and Family Portal remain open to everyone. Since the route gives access to the full `ResidentDetail` page including care notes, medications, and body maps, this merits a deliberate product decision rather than a default.

The room data alignment is semantically coherent: both FacilityMap and CleaningDashboard agree on which rooms exist, which residents occupy them, and which rooms are vacant. They differ in status terminology ("needs-cleaning" vs "needs-attention", "occupied" vs "clean") because they represent different perspectives — facility-management vs cleaning-operations. The risk is silent divergence if either file is edited without updating the other.

## Nav visibility: partial fix leaves Residents exposed

The v1 fix correctly added explicit role arrays to Care Notes, Medications, and Incidents:

```
roles: ['home_manager','deputy_manager','registered_nurse','senior_carer','carer','admin','super_admin','group_admin']
```

Three items remain `roles: null`:

- `/residents` (Residents) — shows all resident data including room numbers, dietary needs, care plans
- `/policies` (Policies) — appropriate for all staff in a CQC-regulated environment
- `/family` (Family Portal) — shows family messages about residents

Since KitchenDashboard already accesses resident dietary data via `useResidents` hook, having operational staff access `/residents` might be intentional. But the route gives access to `ResidentDetail` including care notes, medications, and body maps — clinically-scoped information that operational roles typically wouldn't access. The backend doesn't enforce role-based access control on the residents endpoint either, so this is a frontend-only gate.

## Status vocabulary: defensible but fragile

FacilityMap defines rooms with building-perspective statuses: `occupied`, `vacant`, `needs-cleaning`, `maintenance-required`. CleaningDashboard defines the same 24 rooms with cleaning-perspective statuses: `clean`, `needs-attention`, `overdue`. The mapping is:

```
FacilityMap                    CleaningDashboard
─────────────────────────────  ─────────────────────
occupied (Room 1)         →    clean (Room 1)
needs-cleaning (Room 2)   →    needs-attention (Room 2)
maintenance-required (4)  →    overdue (Room 4)
vacant (Room 8)           →    clean (Room 8)
```

The risk is silent divergence: if someone adds Room 25 to FacilityMap but forgets CleaningDashboard, or changes a resident in one but not the other, the alignment breaks. A shared `ROOM_DATA` constant with per-context status derivation would prevent this.

## KitchenDashboard dietary parsing fragility

KitchenDashboard parses dietary requirements via substring matching (`includes('diabetic')`, `includes('pureed')`) against the `dietary_requirements` field from seeded resident data. This depends on specific string conventions ("Diabetic, fortified drinks") that aren't enforced by the schema. If someone changes the seed data to use different terminology (e.g., "Type 2 diabetes diet") the counts silently drop to zero with no error.
