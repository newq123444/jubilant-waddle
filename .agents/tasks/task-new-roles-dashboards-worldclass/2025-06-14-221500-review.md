# Three operational roles and four reusable widgets for CareVista

This change expands CareVista from clinical and admin roles into operational territory: cleaning, kitchen, and maintenance staff each get dedicated dashboards, demo login accounts, and backend seed data. Four new reusable widget components (WeatherWidget, FacilityMap, StaffWellnessWidget, LiveActivityFeed) are created and wired into relevant dashboards. The implementation follows existing dashboard conventions (inline styles, card-based layouts, simulated data for demo mode). All new code is frontend-only UI with mock data — no new API endpoints or backend logic.

**Watch for:** Navigation visibility gap — new roles can see all `roles: null` nav items (Residents, Care Notes, Incidents, Policies, Family Portal) which may expose care data to non-care staff (confirmed). StaffWellnessWidget is created but never integrated into any dashboard (confirmed). The "Report Concern" form in StaffWellnessWidget has no submit handler — it's a dead button (confirmed).

## High-level view

The role expansion is mechanically complete across all layers — type definitions, login page, dashboard router, seed data, migration enum. The new dashboards are substantial (~300 lines each) with domain-specific widgets (HACCP temperature logs, IPAC compliance zones, PPM calendars) that show genuine understanding of care home operational workflows.

The four reusable components vary in integration. WeatherWidget and LiveActivityFeed land on ManagerDashboard; FacilityMap lands on Cleaning and Maintenance dashboards. StaffWellnessWidget exists but is orphaned — it's never imported or rendered anywhere despite the feature spec calling for integration into CarerDashboard.

The nav system uses `roles: null` to mean "visible to everyone," which means cleaning and kitchen staff see clinical navigation items (Care Notes, Medications, Incidents). Whether this is intentional (all staff need some clinical awareness) or an oversight depends on product intent, but it's worth a deliberate decision rather than a default.

All data in the new dashboards is hardcoded mock arrays. This is consistent with the demo-app nature of the project but means none of the dashboards respond to real state — they'll look identical for every user session, and clearing/completing tasks has no effect.

## Navigation exposure for operational roles

The NAV_ALL array uses `roles: null` to indicate "all roles can see this item." With the addition of cleaning, kitchen, and maintenance roles, these staff now see: Dashboard, Residents, Care Notes, Incidents, Policies, and Family Portal. Some of these make sense (all staff might need Policies), but Care Notes and Incidents expose clinical information that operational staff typically wouldn't access in a CQC-regulated environment.

The fix is either to make those nav items explicit about which roles should see them (`roles: ['home_manager', 'deputy_manager', ..., 'carer']`) or to add a nav-level role group concept. The current implementation leaves the decision implicit rather than deliberate.

## StaffWellnessWidget exists but is never rendered

The FEAT-002 spec calls for integrating StaffWellnessWidget "into CarerDashboard (visible for carer, senior_carer, cleaning, kitchen, maintenance roles)." The component was created (206 lines, fully functional UI) but no dashboard imports it. It's dead code that ships without any rendering path.

## Inconsistent room data between components

CleaningDashboard defines 18 rooms with its own resident assignments. FacilityMap defines 24 rooms with different residents in overlapping room numbers. For example:

- Room 8: "Cyril Newton" in CleaningDashboard, `null` (vacant) in FacilityMap
- Room 11: "Agnes Whitfield, clean" in CleaningDashboard, "Agnes Whitfield, needs-cleaning" in FacilityMap

Since CleaningDashboard embeds FacilityMap, a user sees contradictory information on the same page. The FacilityMap shows Room 2 "needs-cleaning" while the room list below shows Room 2 with a different last-cleaned time and different notes context.

## Report Concern form — inert submit

StaffWellnessWidget renders a "Report a Concern (Confidential)" button that expands to reveal a textarea and Submit button. The Submit button has no `onClick` handler. The textarea is uncontrolled (no state binding). For a demo app, this means clicking Submit does nothing and the form can't be closed after submission — only via the Cancel button. At minimum, submitting should hide the form and display a confirmation message.

## Unused hook calls

```tsx
// CleaningDashboard.tsx
const { data: dash } = useDashboard();
// `dash` is never referenced anywhere in the component

// KitchenDashboard.tsx
const { data: dash } = useDashboard();
// `dash` is never referenced anywhere in the component
```

These fire real API requests (via React Query) on every render of these dashboards. Since the returned data isn't used, they're wasted network calls. Either remove them or wire the generic dashboard stats into the KPI sections.

## KitchenDashboard dietary parsing — works by coincidence

KitchenDashboard parses resident dietary requirements by checking `r.dietary_requirements?.toLowerCase().includes('diabetic')` etc. This depends on the seeded data format for these strings. It works because the demo seed data uses these exact terms, but the approach would silently produce all-zeros if the data format changed. Not a bug today, but worth noting as fragile for a demo that someone might extend.
