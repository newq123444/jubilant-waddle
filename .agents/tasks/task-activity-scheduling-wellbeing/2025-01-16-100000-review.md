# Mobility-gated activity scheduling and holistic wellbeing tracking

Two features land together: an activity scheduling system that enforces mobility-based safety constraints (bed-bound residents can't be enrolled in walking activities), and a broader wellbeing suite covering daily mood/pain/sleep logging, life stories, social isolation alerts, and environment preferences. The combined change adds ~3,300 lines across 19 files spanning two new database migrations, two new backend controllers, full frontend pages with React Query hooks, and seed data.

**Watch for:** No input validation or sanitization on any new endpoint beyond what Postgres CHECK constraints provide (confirmed). The `getEligibleResidents` endpoint builds SQL with string interpolation for the mobility filter rather than parameterized queries — safe in this case since values are hardcoded strings from a switch statement, but the pattern invites future injection if extended (confirmed). The `generateIsolationAlerts` endpoint has an N+1 query pattern iterating residents one-by-one (confirmed). Life story and environment preference endpoints lack `care_home_id` scoping, meaning a staff member could read/write data for residents outside their care home if they know the resident UUID (confirmed).

## High-level view

The mobility enforcement model uses a numeric hierarchy mapping resident statuses to levels, rejecting enrollment when the resident's level falls below the activity's requirement. The check happens server-side at enrollment time but not when a resident's mobility status changes after enrollment, so a resident whose mobility deteriorates remains enrolled in activities they no longer qualify for.

The wellbeing subsystem is four tables — logs, life stories, isolation alerts, environment preferences — each with its own CRUD surface. Isolation alert generation runs on-demand via a manager-triggered POST rather than as a scheduled job, which means alerts go stale unless someone remembers to trigger regeneration. The N+1 loop inside that endpoint will perform poorly at scale.

The life story and environment preference endpoints scope queries by `resident_id` alone without verifying the resident belongs to the requesting user's care home. Since every other endpoint in the codebase scopes by `care_home_id`, this omission creates a cross-tenant data access path.

Activities routes are uniformly gated at `isStaff`, which means any authenticated staff member (kitchen, cleaning, maintenance) can create and delete activities. The acceptance criteria mention an activities coordinator role — the current implementation is more permissive than that.

## Cross-tenant data access on resident sub-resources

The wellbeing controller's `getResidentLifeStory` and `getEnvironmentPreferences` both query solely by `resident_id`:

```typescript
const { rows: [story] } = await query(
  `SELECT ls.* ... FROM resident_life_story ls WHERE ls.resident_id = $1`, [id]
);
```

A staff member at Care Home A who knows a resident UUID from Care Home B can read their life story, environment preferences, and wellbeing logs (the `getResidentWellbeing` endpoint has the same issue — it queries by `residentId` without care home scope). The `updateResidentLifeStory` and `updateEnvironmentPreferences` write paths have the same gap.

The fix is a subquery or join: `WHERE ls.resident_id = $1 AND ls.resident_id IN (SELECT id FROM residents WHERE care_home_id = $2)`.

## Mobility enforcement boundary

Two gaps exist beyond the enrollment-time check:

First, there's no retroactive enforcement. If a resident is enrolled in next week's walking group and their status changes to `bed_bound` today, they stay enrolled. Depending on the care home's workflow this might be intentional (staff review and remove manually) or it might surface a safety incident.

Second, the `createSession` endpoint doesn't validate that at least one resident can participate — a session can be created for an `independent_only` activity even if all residents are bed-bound.

## Input validation absence

None of the new endpoints validate request bodies beyond relying on Postgres CHECK constraints and NOT NULL columns. Examples:

- `createActivity` will accept a `duration_minutes` of -500 (no CHECK in the schema for positive values).
- `logWellbeing` doesn't verify that `residentId` is a valid UUID format before sending it to Postgres.
- `updateResidentLifeStory` passes `importantDates ? JSON.stringify(importantDates) : null` — if `importantDates` is a string instead of an object, it'll be double-serialized.

In a care home system handling personal health data, input validation at the application layer is defense-in-depth against both accidental corruption and injection attempts.

## Isolation alert generation scaling

`generateIsolationAlerts` runs in O(n) queries where n = active residents. It fetches all residents, iterates them, checks `daysSince`, checks for existing alerts, then inserts one by one. The existence check and insert could be collapsed into a single `INSERT ... WHERE NOT EXISTS` with date arithmetic in SQL, eliminating the loop entirely.

## Authorization granularity

Activities routes are gated at `isStaff`, meaning kitchen, cleaning, and maintenance staff can create activities, schedule sessions, add participants, and delete activities. The `deleteActivity` endpoint is `isStaff` rather than `isManager`. If the intent is that only activity coordinators and managers manage the activity library, the middleware needs to be tightened.

## Database schema observations

The `engagement_level` CHECK constraint includes `NULL` as a literal value in the IN list:

```sql
CHECK (engagement_level IN ('high','medium','low','none', NULL))
```

Postgres treats `NULL` in a CHECK constraint as always-true for null values regardless, so the explicit `NULL` is redundant but harmless.

The `wellbeing_logs` table lacks a unique constraint on `(resident_id, log_date)`, which means multiple logs can be created for the same resident on the same day. This might be intentional (shift-based logging) but could lead to confusion in the overview queries that select the latest by date — if two carers log for the same resident on the same day, only one shows in the "today" view depending on ordering.
