# NOT NULL field validation for INSERT handlers

This change adds pre-INSERT validation guards to ~40 handler functions across 36 controller files. Each guard checks required fields from `req.body` before the SQL query executes and returns HTTP 400 with a message listing which fields are required. The approach is uniform: destructure, check falsy, return early. One controller (absence) additionally trims string inputs before validation.

Watch for: **Falsy-value false positives** — several guards use `!value` on numeric fields like `basicHours`, `readingValue`, `doseUnits`, `occupiedBeds`, `amountPence`, `budgetPence`, and rate fields, which will reject legitimate zero values (confirmed). **Incomplete coverage** — the `calculateBradfordScore` function in absence.controller.ts performs an INSERT with `staffId`, `periodStart`, and `periodEnd` from `req.body` but has no validation in this diff (confirmed).

## High-level view

The validation pattern is mechanically simple: a single `if` statement checks `!field` for each required field, then returns a 400 response. This works correctly for string and UUID fields that should never be empty. It breaks for numeric fields where zero is a valid domain value — glucose readings, dose units, hours worked, bed counts, and monetary amounts all have plausible zero cases that would be incorrectly rejected by `!value`.

The `smartHandover.controller.ts` handler is the only one that demonstrates awareness of this problem: it uses `!itemIndex && itemIndex !== 0` to allow zero through. None of the other numeric-field guards apply equivalent logic, creating an inconsistency in the validation approach.

Coverage is broad but not complete. The diff touches every controller that had an INSERT handler and no pre-existing validation. A few INSERT handlers in controllers not touched by this diff (infections, wounds, sleep tracker, weight, continence) already had validation from their initial implementation — those use `throw new AppError(400, ...)` rather than `res.status(400).json(...)`, creating two parallel validation conventions in the codebase. The `calculateBradfordScore` handler in absence.controller.ts has an INSERT but no guard, and the `elearning.createQuiz` validates `moduleId` from `req.params` which Express would never deliver as falsy (it comes from a URL segment).

The inconsistency means the frontend sees two different error response shapes depending on which controller it hits.

## Falsy-value checks on numeric fields

The `!value` idiom works for strings and UUIDs where an empty/null/undefined value is always invalid. For numbers, `!0` evaluates to `true`, meaning a legitimate zero input gets rejected as "missing."

Affected handlers and their problematic fields:

- `staffCosts.recordStaffCost`: `!basicHours` — staff with salaried contracts might report 0 basic hours in a period where only agency hours apply.
- `occupancy.recordOccupancy`: `!totalBeds` and `!occupiedBeds` — zero occupied beds is a valid state (empty care home before opening or after evacuation).
- `billing.createInvoice`: `!amountPence` — a zero-amount invoice (credit note) is a real business scenario.
- `invoicing.createRateUplift`: `!previousRatePence` and `!newRatePence` — zero-to-nonzero rate transition (e.g. NHS-funded resident moving to self-pay) would be rejected.
- `diabetes.logInsulinDose`: `!doseUnits` — might be logged as a "cancelled dose" record.
- `staffCosts.createBudget`: `!budgetPence` — a zero-budget category is unlikely but technically valid.

The `smartHandover.recordAction` handler shows awareness of the problem:

```typescript
if (!itemIndex && itemIndex !== 0) {
  return res.status(400).json({ error: 'itemIndex is required' });
}
```

A cleaner approach for all numeric fields:

```typescript
if (basicHours == null) { ... }  // allows 0, rejects null and undefined
```

## Missing coverage: `calculateBradfordScore`

At line 62 of absence.controller.ts, `calculateBradfordScore` takes `staffId`, `periodStart`, and `periodEnd` from `req.body` and passes them into a SELECT and then an INSERT into `bradford_scores`. If `staffId` is null, the SELECT returns zero spells (harmless) but the INSERT will violate the NOT NULL constraint on `staff_id`. Given that `recordAbsence` in the same file was validated, this gap should be closed.

## Route-param validation is dead code

In `elearning.createQuiz`, the validation checks `!moduleId` where `moduleId` comes from `req.params`. Express routing only invokes this handler when the URL matches `/modules/:moduleId/quiz` (or similar), so `moduleId` is always a non-empty string. The same applies to `menuChoice.updateResidentDietaryProfile` validating `residentId` from `req.params`.

## Two error-response shapes

The new validation returns `{ error: 'field1 and field2 are required' }` directly. Pre-existing validation in other controllers uses `throw new AppError(400, '...')` which propagates to the error-handling middleware. If the middleware serializes errors as `{ message: '...' }` rather than `{ error: '...' }`, the frontend must handle both shapes for 400 responses.
