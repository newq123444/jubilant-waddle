# Black-screen crash fixes and defensive hardening

Three pages (AiCarePlanWriter, ConsentManager, ResidentDigitalTwin) crashed to black when their backend queries failed—typically because optional database columns or tables didn't exist yet. The fix adds `isError` state handling on all affected React Query hooks, null-safe property access throughout the Digital Twin page, a safe JSON/object content parser in the AI Care Plan page, and nested try/catch fallbacks in the backend Digital Twin controller for columns that may not exist. A global `ErrorBoundary` class component wraps all routes so that any future uncaught render crash shows a recovery UI instead of a blank page. The seed script's environmental readings and smart handovers are extended from 7 to 30 days.

**Watch for:**
- The ErrorBoundary's "Try Again" resets state without navigating away—if the crash was caused by corrupted data in React Query cache, the same render will re-fire immediately and re-crash (confirmed).
- The backend fallback pattern silently swallows all query errors, not just "column does not exist"—a genuine connectivity failure would be masked as empty data (confirmed).

## High-level view

The frontend fix destructures `isError` from every affected query hook, shows an error card when true, and adds optional chaining on all nested property access in the Digital Twin page. The Digital Twin page had the deepest exposure because its render tree assumes a fully-populated nested object.

The backend controller adopts a two-tier query strategy: try the full query first, and if it throws (any error), retry with a subset of columns. This makes the Digital Twin API functional regardless of which migration version the database is at, but at the cost of losing error signal—a connection timeout gets the same treatment as a missing column.

The ErrorBoundary sits between `<main>` and `<Routes>`, catching any render error from any routed page. It does not reset the React Query cache or React Router location on recovery, which creates a re-crash risk.

The seed-demo extension generates 30 days of handovers and environmental readings, switching from hourly to every-4-hours sampling after day 7 to keep row count reasonable (~1,500 vs 3,600).

<summary>Issues (4)</summary>

1. **ErrorBoundary reset without cache/route flush** — "Try Again" calls `setState` to clear the error flag but doesn't invalidate the React Query cache or navigate. If the cached query data caused the render crash, the page will crash again immediately. Consider clearing the query client or calling `window.location.reload()`.
2. **Overly broad backend catch masks real failures** — The nested try/catch in `digitalTwin.controller.ts` catches *all* errors, not just `column X does not exist`. A network timeout or permission error would silently produce empty wellbeing data instead of surfacing the failure. Check the error message for the specific Postgres error code before falling back.
3. **AiCarePlanWriter IIFE in JSX** — The `getContent`/`cqc_domains` rendering uses an immediately-invoked function expression inside JSX (`{(() => { ... })()}`). This works but is fragile for future editors—any accidental removal of the surrounding parens or braces breaks the component silently.
4. **No test coverage** — None of the error-handling paths (isError branches, fallback queries, ErrorBoundary catch-and-render) have unit or integration tests. A regression here re-introduces the black screen.

## Backend fallback queries mask failures

The `getDigitalTwin` and `getTimeline` functions both wrap their wellbeing queries in a try/catch that retries with fewer columns, then silently returns `[]` if the retry also fails:

```typescript
let wellbeingTrend: any[] = [];
try {
  // full query with engagement_level, logged_at
} catch (e: any) {
  try {
    // fallback without engagement_level/logged_at
  } catch { /* return empty */ }
}
```

The outer catch doesn't inspect the error. If the database connection drops mid-request, both queries fail, the controller returns an empty `wellbeingTrend`, and the response is a 200 with hollow data. The consumer has no way to distinguish "no wellbeing data exists" from "the database is unreachable."

A narrower guard—checking for Postgres error code `42703` (undefined column) or matching the error message against `does not exist`—would preserve the degradation behavior while still propagating real infrastructure failures.

## ErrorBoundary recovery gap

The ErrorBoundary exposes a "Try Again" button that sets `hasError: false`:

```typescript
handleReset = () => {
  this.setState({ hasError: false, error: null });
};
```

When the state clears, React re-renders `this.props.children`, which re-mounts the crashed page. If the crash was a render-time TypeError on data already resident in the React Query cache (e.g., accessing `.medical.risk_level` on stale partial data), the same access will throw again synchronously, and the boundary catches it again—creating a flicker loop.

The fix is to either call `queryClient.clear()` before resetting (requires passing the query client into the boundary via context) or to force a full page reload with `window.location.reload()`.

## AiCarePlanWriter IIFE pattern

The `getContent` helper correctly handles JSONB dual-representation (pre-parsed object vs JSON string). However, it's invoked via an IIFE inside JSX:

```tsx
{(() => { const content = getContent(selectedPlan); return content?.cqc_domains ? (...) : null; })()}
```

A simple variable extraction above the return (`const content = getContent(selectedPlan)`) followed by a conditional render would be more readable and less prone to accidental breakage during future edits.
