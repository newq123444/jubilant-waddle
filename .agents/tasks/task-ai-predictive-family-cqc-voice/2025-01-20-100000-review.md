# CareVista 4-priority implementation: predictive care, family portal, CQC compliance, voice/SBAR

This change adds four major feature areas to the CareVista care home platform: an AI predictive care engine that scores falls and deterioration risk per resident; an enhanced family portal with AI-generated daily summaries, weekly reports, a photo gallery, and messaging; a CQC compliance automation suite covering five regulatory domains with evidence pack generation and inspection checklists; and voice-to-text care notes with SBAR shift handover generation. Each feature follows the same pattern â€” a database migration, a controller with multiple endpoints, route registration, frontend hooks, and a React page â€” resulting in roughly 5,000 lines of new code touching both frontend and backend.

**Watch for:** SBAR handover sends private/sensitive care notes to the AI without filtering `is_private` (confirmed). Batch analysis endpoints (`runPredictiveAnalysis`, `calculateDomainScores`) make unbounded sequential AI calls proportional to resident or domain count with no timeout protection, risking gateway timeouts and runaway API costs (confirmed). Alerts accumulate without deduplication â€” each analysis run appends new alerts for the same resident/risk even if an active alert already exists (confirmed). The `calculateFallsRisk` handler performs database writes behind a GET route (confirmed).

## High-level view

The predictive care engine's batch analysis path iterates all residents sequentially, issuing ~10 queries per resident per risk type plus AI calls for high-risk cases, with no concurrency limit or progress feedback. Every invocation appends rows to `predictive_risk_scores` and `predictive_alerts` without upsert or dedup logic, so repeated runs pollute the alert list.

The family portal correctly filters private notes from AI-generated summaries (`is_private = FALSE`) and uses upsert semantics for daily/weekly reports. Photo upload stores only a local `/uploads/` path, which requires persistent local storage and a static-file middleware mount.

The CQC compliance suite's `calculateDomainScores` issues ~20 queries across five domains, then conditionally fires up to 5 AI calls. No partial-failure handling exists â€” if the AI call for one domain fails, the entire request 500s and partially-inserted scores leave the UI inconsistent.

The SBAR handover generator queries the full day's care notes without filtering `is_private`, unlike the family summary path. Private notes (safeguarding concerns, sensitive behavioral notes) can surface in handover documents visible to clinical staff. The `calculateDeteriorationRisk` handler is exported but never registered on a route, making it unreachable.

1. **SBAR private note leak** â€” `generateSbarHandover` queries care notes without `is_private = FALSE`, sending private notes to the AI and potentially surfacing them in the handover output. Add the same filter used in `familyPortal.controller.ts`.
2. **Unbounded batch AI calls** â€” `runPredictiveAnalysis` and `calculateDomainScores` make sequential AI API calls proportional to residents/domains with no timeout, concurrency cap, or partial-success handling. 3. **Alert deduplication missing** â€” Each `runPredictiveAnalysis` invocation inserts new alerts unconditionally. Running it twice produces duplicate active alerts for the same resident/risk type. Add an upsert or check for existing active alerts before inserting.
4. **GET route with side effects** â€” `/predictive/residents/:residentId/risk` is a GET but calls `calculateFallsRisk` which inserts into `predictive_risk_scores`. This violates HTTP semantics and can be triggered accidentally by prefetch/crawlers. Change to POST or split into separate calculate (POST) and retrieve (GET) endpoints.
5. **SBAR `residents_covered` always empty** â€” The insert always passes `'{}'` for `residents_covered`. The column and frontend reference it, but the value is never populated from the queried data. Populate from the distinct resident IDs in the aggregated notes.
6. **No partial-failure handling in CQC score calculation** â€” If the AI call for one domain throws, the entire `calculateDomainScores` request fails. Scores computed before the failure are already inserted, but the response never reaches the client, leaving the UI in an inconsistent state. Wrap individual domain AI calls in try/catch.
7. **Predictive score accumulation without cleanup** â€” `predictive_risk_scores` grows unboundedly (every individual and batch calculation appends). Consider a retention policy or deduplication window.

## SBAR handover information boundary

The `generateSbarHandover` controller's first query fetches all care notes for the shift date:

```sql
SELECT cn.content, cn.note_type, cn.is_significant, cn.created_at, ...
FROM care_notes cn
WHERE cn.care_home_id = $1 AND cn.created_at::date = $2 AND cn.deleted_at IS NULL
```

No `is_private` filter exists here, unlike the family portal's `generateDailySummary` which explicitly includes `AND is_private = FALSE`. The full note content is then injected into the AI prompt and the resulting SBAR sections are stored in the database. Clinical staff can view the generated handover, meaning private notes (e.g., safeguarding concerns, sensitive behavioral notes) can be indirectly exposed through the AI's summary text.

## Batch analysis timeout and cost exposure

`runPredictiveAnalysis` iterates all active residents and for each:
1. Calls `calculateRiskInternal` for falls (~5 DB queries, 1 insert)
2. Calls `calculateRiskInternal` for deterioration (~5 DB queries, 1 insert)
3. If either score > 70, inserts an alert
4. If either score > 80, makes an AI API call via `runAiOperation`

For a care home with 60 residents where 10 are high-risk, this means ~600 DB queries + 10 AI calls in a single HTTP request. Express's default timeout is typically 2 minutes; the AI calls alone (at ~3-5s each serially) could exceed that. There is no chunking, no background-job dispatch, and no progress streaming. The same pattern applies to `calculateDomainScores` (up to 5 AI calls) and `generateWeeklyReports` (one AI call per resident).

If the request times out midway, no results reach the client despite scores already being written to the database.

## Alert duplication on repeated analysis runs

The alert insertion in `runPredictiveAnalysis`:

```typescript
if (fallsResult.score > 70) {
  await query(
    `INSERT INTO predictive_alerts (...) VALUES (...)`,
    [careHomeId, resident.id, fallsResult.score, ...]
  );
}
```

There is no `ON CONFLICT` clause, no check for an existing active alert for the same `(resident_id, alert_type)`, and no deactivation of prior alerts before insertion. Running analysis three times for a consistently high-risk resident creates three identical active alerts. The frontend's alerts tab displays all of them.

## GET-triggered writes and unreachable deterioration endpoint

The route `router.get('/predictive/residents/:residentId/risk', isClinical, predictiveCareCtrl.calculateFallsRisk)` binds a handler that inserts into `predictive_risk_scores` on every invocation. Browser prefetch, link-preview bots, or simply navigating to the page creates new database rows.

This endpoint only calculates falls risk despite being the sole `/risk` endpoint. Deterioration risk (`calculateDeteriorationRisk`) is exported but not registered on any route, making it unreachable from the API. The batch `runPredictiveAnalysis` uses an internal helper that covers both risk types, so the individual deterioration endpoint is dead code.

## Voice transcription: hardcoded confidence score

The `transcribeAudio` handler stores a fixed `confidence_score` of `0.95` regardless of actual transcription quality. Since the frontend uses the Web Speech API and sends pre-transcribed text, there is no server-side confidence measurement. This could mislead downstream analytics or quality dashboards that rely on the confidence field. Consider passing the Web Speech API's confidence from the frontend or storing null.

## Family portal authorization scope

All enhanced family portal routes use the `isStaff` guard. There is no mechanism for family members themselves to access the portal. The `sent_to_emails` column in `family_weekly_reports` and the `visibility` column in `family_photo_gallery` suggest external family access was envisaged, but no family-member authentication path exists. If this is intentional (staff-only, sharing via email), the columns are misleading; if family access is planned, it needs its own auth flow.
