# TypeScript error fixes and 8 new care-platform features

This change fixes all pre-existing TypeScript compilation errors in both frontend and backend (missing type declarations, property name mismatches, type assertion gaps), then adds eight new feature modules: AI Care Plan Writer, Predictive Admission Matching, Wellbeing Heatmap, Automated Regulatory Reporting, Smart Handover Intelligence, Digital Consent Manager, Resident Digital Twin, and Environmental Intelligence. Each feature ships as a backend controller with SQL-backed endpoints plus a frontend page with tabbed UI. Both builds pass cleanly.

**Watch for:** The AI Care Plan generator produces hardcoded template text rather than calling any AI service ŌĆö it's a static scaffold labeled as AI output (confirmed). The consent manager route `GET /consents/expiring/all` is shadowed by `GET /consents/:residentId` and will never be reached (confirmed). No database migrations are included, so all new controllers will throw at runtime until 8+ new tables exist (confirmed). The admission matching "AI" scoring is entirely deterministic threshold arithmetic ŌĆö scores will be the same every time regardless of nuance in referral data (confirmed).

## High-level view

The TypeScript fixes are minimal and mostly compile-time-only, though the Incidents.tsx and Training.tsx camelCase corrections fix actual form-binding bugs where inputs were silently rendering as empty strings.

The eight new features follow an identical structural pattern: a backend controller that performs raw SQL queries against assumed tables, returns JSON to the frontend, and a React page component that uses TanStack React Query to fetch and mutate data. None of the features integrate with an actual AI or ML service ŌĆö the "AI" features (care plans, admission matching, handover) generate deterministic output from simple conditionals and database lookups, then label the result as AI-generated.

Route registration is correctly guarded with role middleware (`isManager`, `isClinical`, `isStaff`), though one route shadowing bug exists in the consent manager paths. There are no tests, no migrations, and no API documentation for any of the 30+ new endpoints.

<summary>Issues (7)</summary>

1. **Consent route shadowing** ŌĆö `GET /consents/expiring/all` is registered after `GET /consents/:residentId`. Express matches in registration order, so the parametric route captures "expiring" as a residentId and the expiring-consents endpoint is unreachable. Move `GET /consents/expiring/all` above the parametric route, or rename to `/consents-expiring`.
2. **No database migrations** ŌĆö Eight controllers reference ~10 new tables (`ai_care_plans`, `admission_matches`, `admission_referrals`, `regulatory_notifications`, `smart_handovers`, `handover_actions`, `consent_records`, `capacity_assessments`, `environmental_readings`, etc.) that don't exist. Every endpoint will throw a Postgres "relation does not exist" error at runtime. Add migrations.
3. **AI label on deterministic logic** ŌĆö `aiCarePlan.controller.ts` generates static template text, and `admissionMatching.controller.ts` computes scores with fixed arithmetic. Both present results as "AI-generated" to users. Either integrate an actual LLM call or relabel these as "template-based" / "rule-based" to avoid misleading clinical staff.
4. **Admission matching scores ignore input granularity** ŌĆö The matching algorithm uses 4 hardcoded base scores (85, 80, 90, 75) and subtracts fixed penalties. The `preferences`, `medicalConditions`, and most of the `behavior` payload are accepted but never factored into scores. The feature promises "AI analyzes referrals against home capabilities" but delivers a 4-branch conditional.
5. **Interval query params not validated** ŌĆö `getExpiringConsents`, `getTimeline`, and `getHeatmapHistory` pass user-supplied query params into `($N || ' days')::interval`. Non-numeric input causes a Postgres error that propagates as an opaque 500. Validate that `days`/`hours` are positive integers before query execution.
6. **Environmental correlations join on room_number = zone** ŌĆö `getCorrelations` joins `environmental_readings.zone` to `residents.room_number`. If zones are named differently from room numbers (e.g. "dining_room", "corridor"), the join produces zero rows silently. The correlation feature will appear broken for non-bedroom zones.
7. **No test coverage** ŌĆö 30 files added with zero tests. The controllers contain business logic (scoring algorithms, deadline calculations, heatmap status derivation) that should have unit tests to prevent regression.

## "AI" generation is static templates

The `aiCarePlan.controller.ts` `generateCarePlan` function fetches resident data and recent care notes, then constructs a `carePlanContent` object with hardcoded goal strings:

```typescript
safe: {
  goals: [`Maintain safe environment for ${resident.first_name}`, 'Risk assessments reviewed monthly'],
  interventions: ['Regular safety checks', 'Fall prevention measures in place', 'Medication management protocol'],
}
```

The resident's actual medical conditions, care notes content, and assessment data are gathered but never fed to any language model or inference service. The `recent_observations` field captures 5 notes but they appear only as metadata in the output ŌĆö they don't influence the generated plan text. A care plan generated for a resident with complex dementia needs will contain the same intervention text as one for a resident requiring minimal support.

The `admissionMatching.controller.ts` follows the same pattern: it starts from fixed scores (`capabilityScore = 85, staffScore = 80, roomScore = 90, compatibilityScore = 75`) and applies at most 4 penalty branches. The `preferences` and `medicalConditions` fields from the request body are captured in JSON but have zero effect on the computed score.

## Consent manager route shadowing

The route registrations in `routes/index.ts`:

```typescript
router.get('/consents/:residentId',            isStaff, consentManagerCtrl.listConsents);
router.patch('/consents/:id',                  isStaff, consentManagerCtrl.updateConsent);
router.get('/consents/expiring/all',           isManager, consentManagerCtrl.getExpiringConsents);
```

Express matches routes in registration order. `GET /consents/expiring/all` will never be reached because `GET /consents/:residentId` matches first with `residentId = "expiring"`. The query against `consent_records WHERE resident_id = 'expiring'` will return an empty array rather than the intended expiring-consents list ŌĆö a silent failure that masks the feature entirely.

## Missing database schema

Every new controller issues `INSERT INTO` or `SELECT FROM` against tables that exist nowhere in the codebase or migration history:

- `ai_care_plans` (aiCarePlan.controller.ts)
- `admission_matches`, `admission_referrals` (admissionMatching.controller.ts)
- `regulatory_notifications` (regulatoryReporting.controller.ts)
- `smart_handovers`, `handover_actions` (smartHandover.controller.ts)
- `consent_records`, `capacity_assessments` (consentManager.controller.ts)
- `environmental_readings` (environmentalIntel.controller.ts)

The wellbeing heatmap and digital twin controllers query existing tables (`wellbeing_logs`, `residents`, `care_notes`, `incidents`, `medications`) so they'll work against the existing schema. The other six will fail immediately at runtime.

## Wellbeing heatmap defaults to "green" for unassessed residents

The heatmap controller assigns traffic-light status through a cascade of `if/else if` checks on mood and pain level. A resident with no wellbeing log at all (all fields null from the `LEFT JOIN LATERAL`) defaults to `green`/"Settled", which masks residents who haven't been assessed recently. There's no "unknown" or "stale data" status ŌĆö a room with no observations for a week looks identical to one where the resident was just logged as happy.

## Incidents and Training fixes are behavioral corrections

The camelCase fixes in Incidents.tsx and Training.tsx are not just type fixes ŌĆö they correct actual bugs. The form `value` attributes previously read `form.resident_id`, `form.incident_type`, `form.staff_id`, `form.course_name` etc., but the state object defined these as `residentId`, `incidentType`, `staffId`, `courseName`. The inputs were always rendering with `value={undefined}` (uncontrolled) despite the `onChange` handlers correctly writing to the camelCase properties. This means those form fields appeared to work but lost their displayed value on re-render.
