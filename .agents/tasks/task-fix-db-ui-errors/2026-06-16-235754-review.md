# Database schema, controller SQL, and frontend API path alignment

This change creates a new migration (014) adding 12 missing tables and several ALTER TABLE statements, then fixes SQL column/table name mismatches across 6 backend controllers, reorders routes to prevent a parameter collision, adds input validation to the absence controller, and corrects 6 frontend API path mismatches plus a related TypeScript signature fix.

Watch for:
- **Migration-only schema strategy** — the 12 new tables and columns exist only as a SQL file in `backend/migrations/`. There is no automated migration runner, so the user's database won't have these tables until the migration is manually applied. The user's live runtime errors (resident_belongings, resident_weights, engagement_level) confirm this gap. (confirmed)
- **Route collision only partially fixed** — `/wellbeing/heatmap` is now correctly placed before `/:residentId`, but when the heatmap handler errors and calls `next(err)`, Express falls through to `getResidentWellbeing` producing a second error. The user's logs show `"heatmap"` being passed as a UUID. (confirmed)
- **Medication discontinue is now a destructive DELETE** — the frontend changed from PATCH (soft discontinue with reason) to DELETE with no body, which likely hard-deletes the medication record rather than marking it inactive. (likely)

## High-level view

The migration file is structurally sound — all 12 tables use UUID PKs, care_home_id FKs, appropriate indexes, and IF NOT EXISTS guards. The column additions to wellbeing_logs (engagement_level, logged_at) directly address the runtime errors the user reports, but only once the migration is actually executed against the database. No application-level migration runner or startup check exists to make this automatic.

The route reordering is the right fix for the `/wellbeing/heatmap` collision, though the user's error log reveals it doesn't fully prevent the fallthrough scenario when the heatmap handler itself throws (which it currently does because the migration hasn't been applied).

The frontend API path corrections align client calls with actual backend route registrations. The one behavioral change (PATCH-to-DELETE for medication discontinue) goes beyond a path fix — it changes semantics from "update status" to "destroy resource," silently discarding the discontinuation reason.

## Route collision and heatmap fallthrough

The fix moves the three `/wellbeing/heatmap*` routes above `/wellbeing/:residentId`. Without this, the heatmap routes would never execute. However, the user's runtime log shows two errors per request to `/api/wellbeing/heatmap`:

1. `column wl.engagement_level does not exist` — from `getHeatmap`
2. `invalid input syntax for type uuid: "heatmap"` — from `getResidentWellbeing`

The second error occurs because `getHeatmap` calls `next(err)`, and Express proceeds to the next route handler matching the same path pattern (`/:residentId`) before the error middleware handles it. The double-error will persist until either (a) the migration is applied (fixing the root cause so the heatmap handler succeeds) or (b) the heatmap handler responds directly with the error instead of calling `next(err)`, or (c) the `:residentId` handler validates UUID format and short-circuits.

## Migration: logged_at column lifecycle

`logged_at` is added as a nullable column with no DEFAULT:
```sql
ALTER TABLE wellbeing_logs ADD COLUMN IF NOT EXISTS logged_at TIMESTAMPTZ;
UPDATE wellbeing_logs SET logged_at = created_at WHERE logged_at IS NULL;
```

The backfill handles existing rows. But the `logWellbeing` controller (which inserts new wellbeing_logs rows) would need to supply `logged_at` explicitly on every INSERT, or the column needs `DEFAULT NOW()`. Every heatmap query uses `ORDER BY wl.logged_at DESC` and `WHERE wl.logged_at > NOW() - interval`, so NULL values would silently disappear from results.

## Medication discontinue semantics change

```typescript
// Before
discontinue: (id: string, data: object) => api.patch(`/medications/${id}/discontinue`, data),

// After
discontinue: (id: string) => api.delete(`/medications/${id}`),
```

This changes from a PATCH with a body (setting `active = false` with a reason) to a DELETE with no body. In a care home context, medication audit trails are regulatory requirements — a soft-delete (setting active=false with a discontinued_at timestamp and reason) is the expected pattern.

The hook signature preserves `data?: object` in the destructured input but never passes it to the API call:
```typescript
mutationFn: ({ id }: { id: string; data?: object }) => emarApi.discontinue(id).then(r => r.data)
```

Existing call sites passing `{ id, data: { reason: '...' } }` will type-check but silently discard the reason.

## Frontend API path corrections

Six API paths are realigned to match backend route registrations. The `acknowledgeRequest` call also changes from `/requests/${id}/acknowledge` (sub-resource action) to `/requests/${id}` (PATCH on the resource itself), which is semantically different but matches the backend's single PATCH handler.

The `residentTabletApi.getResidentView` changes from `/resident-tablet/${residentId}/view` to `/tablet/residents/${residentId}` — a different resource hierarchy that matches the actual backend registration.
