# Ten resident quality-of-life features for CareVista

This change adds ten self-contained modules to the care home management platform: Personal Music Therapy, Menu Choice System, Friendship Mapper, Daily Purpose Planner, Mood-Responsive Environment, Digital Photo Frame Feed, Sleep Quality Tracker, Intergenerational Programme Manager, Rehabilitation Goal Tracker, and Birthday & Celebration Planner. Each module follows the same structural pattern: a SQL migration, an Express controller with raw SQL queries, route registration with role-based guards, frontend TypeScript types, an API service layer, React Query hooks, and a page component wired into the sidebar. The approach is consistent across all ten features and the database schema is well-normalized with appropriate indexing.

**Watch for:** Pervasive frontend-to-backend API path mismatches that will cause 404s at runtime (confirmed). No server-side input validation on any of the 83 new endpoints (confirmed). The celebrations birthday-detection query will fail for residents born on dates that don't exist in non-leap years (likely).

## High-level view

The frontend API service layer and the backend route definitions disagree on URL paths for the majority of the ten features. The Music Therapy frontend calls `/music-therapy/residents/:id/preferences` while the backend registers `/music-therapy/preferences/:residentId`. Photo Frame uses `/photo-frames/` (plural with 's') in the frontend but `/photo-frame/` (singular) in the backend. Sleep Tracker, Friendship Mapper, Mood Environment, and Rehab Goals all exhibit similar misalignment. Every HTTP call from these pages will hit an unmatched route and return a 404.

None of the ten controllers perform any input validation before constructing SQL queries. Fields like `moodBefore`, `qualityScore`, `celebrationType` are extracted from the request body and passed directly to parameterized queries. While SQL injection is prevented by parameterization, malformed or out-of-range values will propagate to the database (CHECK constraints will throw unformatted Postgres errors) or silently insert garbage. This is a care environment where data integrity matters clinically.

The `getUpcoming` celebrations controller builds a next-birthday date using `MAKE_DATE` with the birth day and month in the current year. For a resident born on February 29, this will throw a Postgres error in non-leap years because `MAKE_DATE(2025, 2, 29)` is invalid.

The frontend type definitions and the backend column/field naming show substantial drift. The frontend types define fields like `preference_level`, `specific_artists`, `song_title`, `disturbances_count` while the database columns are `tempo_preference`, `preferred_artists`, `title`, `disturbances`. The TypeScript interfaces won't cause runtime failures on their own, but they represent an incorrect contract that will surface as undefined fields in the UI.

## Pervasive frontend-backend route mismatch

A systematic comparison of the backend route registrations against the frontend API service reveals that 8 of 10 features have path disagreements. The specific mismatches:

**Music Therapy:** Backend registers `GET /music-therapy/preferences/:residentId` and `GET /music-therapy/sessions/:residentId`. Frontend calls `GET /music-therapy/residents/:residentId/preferences` and `GET /music-therapy/residents/:residentId/sessions`. Backend `PATCH /sessions/:id/end` vs frontend `POST /sessions/:id/end`. Backend `GET /effectiveness` (query param) vs frontend `GET /effectiveness/:residentId`.

**Menu Choices:** Backend `GET /menu-choices/dietary-profile/:residentId` (singular) vs frontend `GET /menu-choices/dietary-profiles/:residentId` (plural). Backend `POST /menu-choices` vs frontend `POST /menu-choices/submit`.

**Friendship Mapper:** Backend `GET /friendship-mapper/connections/:residentId` vs frontend `GET /friendship-mapper/residents/:residentId/connections`. Backend `GET /friendship-mapper/isolated` vs frontend `GET /friendship-mapper/isolated-residents`.

**Mood Environment:** Backend `GET /mood-environment/history/:residentId` vs frontend `GET /mood-environment/residents/:residentId/history`. Backend `GET /mood-environment/effectiveness` (query param) vs frontend `GET /mood-environment/effectiveness/:residentId`.

**Photo Frame:** Backend uses singular `/photo-frame/` throughout. Frontend uses plural `/photo-frames/`. Backend uses `PATCH` for approve/reject. Frontend uses `POST`. Backend `GET /photo-frame/photos/:residentId` vs frontend `GET /photo-frames/photos` with query params. Backend `GET /photo-frame/viewing-history/:residentId` vs frontend `GET /photo-frames/residents/:residentId/history`.

**Sleep Tracker:** Backend `POST /sleep-tracker/logs` vs frontend `POST /sleep-tracker/log`. Backend `GET /sleep-tracker/history/:residentId` vs frontend `GET /sleep-tracker/residents/:residentId/history` (and same pattern for profile, disturbances, suggestions).

**Rehab Goals:** Backend `POST /rehab-goals` vs frontend `POST /rehab-goals/goals`. Backend `PATCH /rehab-goals/milestones/:id` vs frontend `POST /rehab-goals/milestones/:id/progress`. Backend `PATCH /rehab-goals/:id/celebrate` vs frontend `POST /rehab-goals/milestones/:id/celebrate`. Backend `GET /rehab-goals/report/:residentId` vs frontend `GET /rehab-goals/residents/:residentId/report`.

**Intergenerational:** Backend `PATCH /intergenerational/visits/:id/outcome` vs frontend `POST /intergenerational/outcomes`.

**Celebrations:** Backend `POST /celebrations/tasks` vs frontend `GET /celebrations/:id/tasks`. Backend `PATCH /celebrations/tasks/:id/complete` vs frontend `POST /celebrations/tasks/:id/complete`.

Only Purpose Planner and the celebrations GET endpoints are correctly aligned.

## Leap-year birthday detection failure

In `celebrations.controller.ts`, the `getUpcoming` function computes next birthdays:

```sql
MAKE_DATE(EXTRACT(YEAR FROM CURRENT_DATE)::INT,
          EXTRACT(MONTH FROM date_of_birth)::INT,
          EXTRACT(DAY FROM date_of_birth)::INT)
```

`MAKE_DATE(2025, 2, 29)` throws `ERROR: date field value out of range`. This will cause the entire `getUpcoming` endpoint to 500 for any care home with a resident born on Feb 29.

## Friendship strength never decays

In `friendshipMapper.controller.ts`, `recordObservation` increments the connection strength unconditionally:

```typescript
const newStrength = Math.min(10, existing[0].strength + 1);
```

There is no time-based decay, no mechanism to reduce strength for negative observations, and no way for strength to decrease at all. A resident pair that had ten positive interactions six months ago and nothing since will still show maximum connection strength. The "isolated residents" detection (connections < 2) will never fire for residents who had historical interactions, even if those interactions stopped months ago.

## Sleep suggestions grouping by JSONB equality

In `sleepTracker.controller.ts`:

```sql
GROUP BY interventions
```

PostgreSQL JSONB equality is order-sensitive for arrays. `["a","b"]` and `["b","a"]` are different values, so identical intervention sets in different orders produce separate groups. The suggestions will fragment, reducing their apparent efficacy.

## Admission anniversary detection is incorrect

The `getUpcoming` celebrations query filters anniversaries by:

```sql
AND EXTRACT(DAY FROM admission_date) BETWEEN EXTRACT(DAY FROM CURRENT_DATE) AND EXTRACT(DAY FROM CURRENT_DATE) + $2
```

`$2` is `lookAheadDays` which defaults to 30. If today is the 5th, this becomes `BETWEEN 5 AND 35`. Since no month has day 35, this works but only catches anniversaries in the current month. If today is the 28th, `BETWEEN 28 AND 58` only catches days 28-31 (or 28-30, 28-29) of the current month. It completely misses anniversaries in the following month regardless of the `lookAheadDays` parameter.

## No input validation across 83 endpoints

The controllers extract fields from `req.body` and `req.query` without any validation:

```typescript
const { residentId, moodBefore, notes } = req.body;
```

If `moodBefore` is a string like `"happy"` rather than an integer 1-10, the Postgres CHECK constraint will reject it with an unformatted database error bubbling to the client. If `residentId` is missing entirely, the INSERT will fail with a NOT NULL violation. Every endpoint in the change has this characteristic. The existing codebase shares this pattern (no validation library is present), but the scale of 83 new endpoints makes this worth addressing.
