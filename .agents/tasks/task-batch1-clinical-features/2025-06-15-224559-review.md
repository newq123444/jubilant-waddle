# Eight clinical features for CareVista: NEWS2, wounds, infections, continence, rota, search, risk scores, and medication interactions

This change adds eight clinical/operational features spanning the full stack: a database migration with 12 new tables, 8 backend controllers with route registration, 8 frontend pages with types, API services, and React Query hooks. The approach follows the existing CareVista patterns consistently — Express controllers with `care_home_id` tenant isolation, React pages with inline styles, React Query hooks for data fetching. The implementation is substantial (5700+ lines) and structurally sound, but ships several request payload mismatches between frontend and backend that will produce 400 errors or undefined values at runtime.

**Watch for:** Frontend-to-backend field name mismatches in NEWS2 calculator will break submissions (confirmed). Continence overview API path mismatch will 404 (confirmed). NEWS2 escalation default status value violates the DB CHECK constraint (confirmed). Frontend SpO2 Scale 2 scoring is clinically incorrect (confirmed). NL search time range uses string interpolation into SQL — safe today but fragile against future changes (likely).

## High-level view

The migration introduces well-structured tables with appropriate CHECK constraints, UUID primary keys, foreign keys for tenant isolation, and indexes on common query paths. One notable gap: the `continence_assessments` table lacks a UNIQUE constraint on `(care_home_id, resident_id)`, forcing the controller into a check-then-insert/update pattern that's susceptible to race conditions.

The NEWS2 calculator implements the NHS scoring algorithm on both frontend (for real-time preview) and backend (for persistence), but the implementations diverge on SpO2 Scale 2 and the frontend sends snake_case field names while the backend expects camelCase. This means every NEWS2 submission from the UI will fail.

The medication interaction checker uses a hardcoded database of 17 interaction pairs and performs an O(n^2 * rules) check against each medication pair per resident. For typical care home residents (5-15 medications), this is acceptable, but the per-pair `SELECT` inside nested loops means each check fires potentially hundreds of queries.

The smart rota generator implements a greedy round-robin assignment respecting the 48-hour working time directive and contract hours, but always assigns the same staff in the same order, meaning the first nurses/carers in the query result always get shifts while later ones may get none.

The natural language search parser handles falls, medications (by name and category), conditions, risk levels, room numbers, and time ranges. The time range is interpolated directly into SQL, but the values are constrained to a fixed internal set so it's not exploitable today.

The continence assessment frontend calls `/continence/home-overview` but the backend registers `/continence/overview`, so the overview dashboard will 404 in production.

3. **NEWS2 escalation default status value** — `respondToEscalation` uses `status || 'responded'` but the DB CHECK allows only `'pending'`, `'acknowledged'`, `'resolved'`. If no status is provided by the caller, the INSERT will fail the constraint. Fix: default to `'acknowledged'`.
4. **Frontend SpO2 Scale 2 scoring is clinically wrong** — The frontend `scoreSpo2` returns 3 (highest score) for SpO2 >= 97 on supplemental oxygen, which makes no clinical sense. The NHS Scale 2 scores SpO2 93-94 on target as 0, 95-96 as 1, >= 97 as 3 (because over-oxygenation in COPD patients is harmful). Verify against the official NHS chart and align with the backend implementation.
5. **Continence pad_status query mismatch** — `getHomeOverview` filters `pad_status IN ('changed', 'wet', 'soiled')` but 'changed' is not in the DB CHECK constraint (`'dry','wet','soiled','not_applicable'`). The pad change count will under-count. Fix: use `event_type = 'pad_change'` instead.
6. **Smart rota greedy assignment bias** — Staff are assigned in query result order (alphabetical by last name), so early-alphabet staff always fill first and get maximum hours while later staff get fewer shifts. Fix: randomize or rotate assignment priority.
7. **Medication interaction O(n^2) per-pair DB query** — `checkInteractions` runs a SELECT inside a triple-nested loop (meds * meds * rules). For a resident with 10 meds and 17 rules, that's up to 765 queries per check. Fix: batch the existence check or pre-fetch all existing interactions for the resident.
8. **Continence assessment upsert race condition** — The check-then-insert/update pattern in `createAssessment` allows duplicate assessments if two requests arrive simultaneously. Fix: add a UNIQUE constraint on `(care_home_id, resident_id)` and use `ON CONFLICT`.
9. **NL search time range SQL interpolation** — `INTERVAL '${parsedIntent.timeRange}'` is safe today because `parseQuery` only produces from a fixed set, but if the parser is extended to accept user-provided durations, this becomes an injection vector. Fix: parameterize using `($n || ' days')::INTERVAL` pattern already used elsewhere in the codebase.

## NEWS2 frontend-backend contract mismatch

The frontend `News2Calculator.tsx` submits the mutation payload with snake_case keys:

```typescript
calculateMutation.mutate({
  resident_id: selectedResident,
  respiratory_rate: respiratoryRate,
  supplemental_oxygen: supplementalOxygen,
  systolic_bp: systolicBP,
  // ...
});
```

The backend `news2.controller.ts` destructures camelCase from `req.body`:

```typescript
const { residentId, respiratoryRate, supplementalOxygen, systolicBp, pulse, consciousness, temperature, notes } = req.body;
```

Since JavaScript object destructuring is case-sensitive, `residentId` will be `undefined` when the body contains `resident_id`. The validation check `if (!residentId || respiratoryRate == null ...)` will throw a 400 on every submission from the UI. The same mismatch affects all vital sign fields.

Additionally, the frontend implements SpO2 Scale 2 (supplemental oxygen) with an inverted scoring curve. The function returns 3 for SpO2 >= 97, which represents the highest clinical concern. In the actual NHS NEWS2 Scale 2, this is correct for COPD patients (over-oxygenation is dangerous), but the intermediate ranges (<=83: 3, <=85: 2, <=87: 1, 88-92: 0, 93-94: 1, 95-96: 2) don't match any published NEWS2 chart. The backend's Scale 2 implementation is simpler (>= 97: 0, 95-96: 1, 93-94: 2, <= 92: 3) which is actually the standard Scale 1, not Scale 2. Neither implementation correctly represents the NHS NEWS2 Scale 2 chart.

## Continence overview route and pad_status mismatch

Two distinct issues affect the continence overview feature:

The frontend `api.ts` defines `getOverview: () => api.get('/continence/home-overview')` but the backend route is registered as `router.get('/continence/overview', isManager, continenceCtrl.getHomeOverview)`. This 404 will be invisible during development without a running backend.

Separately, in `getHomeOverview`, the pad change count query uses:
```sql
AND pad_status IN ('changed', 'wet', 'soiled')
```

But the `continence_logs.pad_status` CHECK constraint only allows `'dry', 'wet', 'soiled', 'not_applicable'`. The value `'changed'` can never exist in the database. The intent was probably to count pad change events, which should use `event_type = 'pad_change'` from the `event_type` column instead.

## Smart rota generation strategy

The rota builder uses a deterministic greedy algorithm: for each day and shift type, it iterates through nurses and carers in query result order (alphabetical by `last_name, first_name`) and assigns until the minimum staff requirement is met or hours are exhausted. This produces functional rotas that respect the 48-hour working time directive, but with consistent bias: staff appearing first alphabetically will always be preferentially assigned.

The generator also doesn't account for rest periods between shifts. A staff member ending a night shift at 07:00 could be assigned a day shift starting at 07:00 the next day without the legally required 11-hour rest period between shifts (Working Time Regulations 1998, Regulation 10).

The `publishRota` function copies generated shifts to the main `shifts` table, casting `shift.shift_type` directly to `shift_type::shift_type`. If the generated shift types ('day', 'evening', 'night') don't match the enum values in the existing `shift_type` PostgreSQL type, this will fail at runtime.

## Medication interaction checker performance

The `checkInteractions` endpoint fetches all active medications for a resident, then for every pair (i, j) where i < j, iterates all 17 rules. For each match, it runs a SELECT to check if an active record already exists. For a resident with N medications:
- Pairs checked: N*(N-1)/2 * 17 rule comparisons
- Potential DB queries: up to N*(N-1)/2 existence checks + inserts

With 10 medications: 45 pairs * 17 rules = 765 comparisons, up to 45 SELECT queries plus inserts. For a care home with 60 residents, running a batch check would be 2700+ queries. This won't cause timeouts for individual checks but would if batch operations are added later.

## NL search: safe but fragile SQL interpolation

The `nlSearch.controller.ts` `search` function interpolates `parsedIntent.timeRange` directly into SQL:

```typescript
sql += ` AND i.incident_date > NOW() - INTERVAL '${parsedIntent.timeRange}'`;
```

The `parseQuery` function only ever sets `timeRange` to one of four fixed values ('1 month', '7 days', '1 year', '3 months'), so this is not currently exploitable. However, the pattern diverges from the rest of the codebase which uses `($n || ' days')::INTERVAL` with parameterized queries. If a future developer adds user-specified time ranges without realizing the interpolation pattern, it becomes an injection vector.

## Missing test coverage

The project has no automated tests (validation is via TypeScript compilation). For a healthcare application implementing clinical scoring algorithms (NEWS2, Waterlow, MUST, Falls), the absence of unit tests for scoring logic is a significant gap. The NEWS2 Scale 2 discrepancy between frontend and backend demonstrates exactly the kind of bug that scoring unit tests would catch. The Waterlow scoring uses data from multiple tables (weights, wounds, continence logs, care notes, medications) and produces a composite score — any regression in the underlying queries would silently produce incorrect risk levels.
